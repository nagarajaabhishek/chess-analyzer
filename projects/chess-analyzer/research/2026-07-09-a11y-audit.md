# ChessNow Web — WCAG 2.2 AA Accessibility Audit

> Item 18 (code portion). Automated + manual code audit of the ChessNow web client
> run against the live app on 2026-07-09. The moderated screen-reader sessions that
> item 18 also calls for are a **human dependency** — the checklist at the end is
> what to run once testers are lined up. Doing the code audit first means those
> sessions surface real UX issues, not bugs we already knew about.

## Why this matters
The FAQ and marketing already claim the site is "fully optimized for screen readers
like JAWS and NVDA" and the product targets blind and low-vision players. That claim
has to be earned in code before it's earned in press. This pass closes the
code-level gaps; the human sessions validate the lived experience.

## Fixed this pass (committed)
| # | Finding | WCAG | Fix |
|---|---------|------|-----|
| 1 | `.use-case-note` text was **3.42:1** on the card (fails AA for normal text) | 1.4.3 | Recolored to `--text-2` → **8.61:1** |
| 2 | Source toggle used `role="tab"`/`tablist` **without** the tab-widget contract (no roving tabindex, arrow keys, or `aria-controls`) — a false ARIA promise a screen reader announces but can't fulfil | 4.1.2 | Switched to `role="group"` + `aria-pressed` toggle buttons — honest single-select, fully keyboard-operable via native buttons |
| 3 | `#personality-select` (Thara commentary dropdown) had no programmatic label | 1.3.1 / 4.1.2 | Added `sr-only` `<label for>` |
| 4 | Continuous decorative animations (voice-indicator bars, hero glow, mic pulse) ran regardless of the OS reduce-motion setting | 2.2.2 / 2.3.3 | Global `@media (prefers-reduced-motion: reduce)` safety net |

## Verified already-good (no change needed)
- Single `<h1>`, one `<main>`, one `<nav>`, and a working "Skip to main content" link.
- **0** images without `alt`; **0** buttons/links without an accessible name.
- All form controls now have labels (`username-input`, `pgn-input`, `difficulty-select`, `personality-select` via `sr-only`; `chk-show-arrows` via a wrapping `<label>`).
- The interactive demos are surprisingly strong: the "speak your first move" board announces each move through an `aria-live="polite"` status region, the reveal button carries `aria-expanded`/`aria-controls`, and the public-stats strip is an `aria-live` region.
- Body/heading text and the gold-on-dark accents pass AA (inactive tab & search-sub both 7.71:1).

## Still open — needs human screen-reader testing (item 18 human dependency)
These can't be fully validated by code inspection; schedule moderated sessions
(JAWS + NVDA on Windows, VoiceOver on iOS/Capacitor, plus at least one blind or
low-vision participant per the target segment):

1. **The analysis board** — chessboard-js renders piece `<img>`s; confirm a SR user can actually *read the position*, not just tab past it. May need a board-to-text region or per-square labels. **Highest-risk item.**
2. **"Speak your first move" demo end-to-end** with a real SR + mic — does the live-region cadence work when the user is also dictating? Test the tappable-chip fallback path under SR.
3. **Voice-call overlay** (browser SpeechRecognition play) — focus management when the modal opens/closes, and whether call state changes are announced.
4. **Dynamic view switches** (SPA `showView`) — is focus moved to the new view's heading? Today it likely isn't; verify and add focus management if SR users get lost.
5. **The share-review-card** flow — the generated canvas is an image; confirm the "downloaded/shared" outcome is announced.
6. Full keyboard-only walkthrough of every flow (no mouse), and a 200%-zoom / 400%-reflow pass.

## Recommended next code work (post-sessions, not done here)
- Add a textual board description region for the analyzer (drives #1 above).
- Move focus to the `<h1>`/heading of each SPA view on navigation (#4).
- Audit focus trapping + return focus for the voice-call overlay (#3).
